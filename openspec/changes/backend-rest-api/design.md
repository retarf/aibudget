## Context

The aibudget app currently has no backend. This change introduces the first
REST API and persistence layer for the core domain — budgets, transactions, and
categories — and runs it under Docker Compose so the whole environment starts
with one command. The proposal defines three capabilities: `budget-api`,
`transaction-api`, and `category-api`.

## Goals / Non-Goals

**Goals:**

- A `docker-compose.yml` at the project root running two services: the FastAPI
  backend and a PostgreSQL database.
- A `backend/` package with a clear layering: routers → services → models.
- SQLAlchemy ORM models for budgets, transactions, and categories, with Pydantic
  schemas kept separate from the ORM models for request/response validation.
- An initial database schema created on startup.
- The stack managed via the existing `cli compose` commands.

**Non-Goals:**

- Authentication / authorization.
- Reports and the NATS microservices.
- A production-grade deployment setup (TLS, secrets management, scaling).
- A full migration tool — see Risks below.

## Decisions

- **Layering.** `backend/` is split into `routers/` (HTTP only — parse, call a
  service, return a schema), `services/` (business rules and validation), and
  `models/` (SQLAlchemy ORM). Route handlers stay thin.
- **Schemas vs models.** Pydantic schemas (`schemas/`) define the API contract
  with separate `*Create` / `*Read` shapes; SQLAlchemy models define
  persistence. They are never the same class.
- **Database.** PostgreSQL runs as a Compose service; its data is kept on a
  named volume so it survives `cli compose down`. The backend connects via a
  `DATABASE_URL` environment variable.
- **Environment configuration.** All credentials and environment variables live
  in a `.env` file at the project root. `.env` is created, edited, and deleted
  **manually by the developer** — it is never generated by tooling and never
  committed. A committed `.env.template` lists every required variable with
  example values; developers copy it to `.env` and fill in real values.
  `docker-compose.yml` reads its environment from `.env`.
- **Driver / sessions.** SQLAlchemy 2.x with the `psycopg` (v3) driver. A
  session-per-request dependency provides the DB session to routers.
- **Domain rules** (enforced in the service layer, matching the specs):
  - A budget's end date must be strictly after its start date.
  - A transaction's date must fall within its budget's period; amount > 0.
  - Category names are unique per kind; a category in use cannot be deleted.
  - Deleting a budget cascades to its transactions.
- **Schema creation.** For now the backend creates tables on startup
  (`create_all`). No migration tool yet.
- **Compose service names.** `backend` and `db`, so `cli compose logs backend`
  and `cli compose rebuild backend` work as documented.

## Risks / Trade-offs

- **No migrations.** `create_all` on startup is simple but cannot evolve an
  existing schema. Acceptable while the schema is new; a migration tool (e.g.
  Alembic) will be needed before real data exists.
- **`create_all` race.** If the backend starts before PostgreSQL is ready,
  startup fails. Mitigated with a Compose healthcheck on `db` and `depends_on`
  with `condition: service_healthy`.
- **Cascading deletes.** Deleting a budget removes its transactions; this is per
  spec but is destructive and irreversible. Documented behavior, not a bug.
- **Synchronous stack.** `psycopg` (sync) with FastAPI is simpler than async but
  caps throughput. Acceptable at this stage; revisitable if load demands it.
